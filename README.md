# OzonTestTask
## Инструкция для запуска
1. Выполнить скрипт для создания таблиц БД
2. В 'OzonTestTask/appsettings.json' и в 'ReportWorker/appsettings.json' указать актуальные строки подключения в секции 'ConnectionStrings'
3. Запустить командную строку и перейти в директорию с проектом, после чего прописать команду 'docker-compose up -d'

## Описание проделанной работы
Был создан контроллер 'api/reports' для работы с отчетами по конверсии. Для данного контроллера было создано три роута:
- POST 'create-request' - отвечает за создание запроса на отчет
- GET 'check-request/{id}' - отвечает за проверку статуса отчета по его идентификатору
- GET 'report/{id}' - отвечает за получение отчета по его идентификатору

Кроме того, был создан gRPC сервис для получения отчета через gRPC, однако, его не получилось запустить в докере, хотя попытки были, при обычном запуске проекта сервис работает.
Для обработки создаваемых запросов был создан worker, являющийся фоновой службой, который через Kafka получает идентификаторы запросов на обработку и производит с ними необходимые действия.
Логика у всего сервиса простая - пользователь создает запрос на получение отчета по конверсии. Если по интересующим его параметрам уже существует отчет, то в ответ он получает его идентификатор. Если же такого отчета нет, то проверяются уже существующие запросы. В случае совпадения - возвращается идентификатор запроса с такими же данными. В случае, если запрос является уникальным, для него создается соответствующая запись, а пользователь получает ее идентификатор, параллельну на шину сообщений Kafka отправляется также идентификатор созданного отчета. Отчеты обрабатываются воркером в фоновом режиме при их поступлении по шине. По описанным выше контроллерам пользователь с помощью идентификаторов может либо проверить статус выполнения запроса, либо же получить уже сформированный отчет.
Проект был покрыт небольшим количеством тестов, а также был помещен в докер образ. С последним возникла проблема, т.к. это мой первый именно серьезный опыт создания докер образа. До этого мне не призодилось заставлять докер образ работать с Kafka, что и вызвало особенные трудности, про gRPC сервис вообще молчу.
Само задание показалось очень интересным, единственное, я не до конца понял формулировку 'идентификатор оформления'. Я спросил у рекрутера, но она не смогла мне помочь. Для себя я решил, что это идентификатор способа оформления действия(мобильное приложение, браузер и т.д.), т.к. это может быть реально близко к конверсии. Был еще вриант, что это идентификатор оформления заказа(покупки) или его статуса, если он был куплен(лежит в корзине, оплачен), но это не так хорошо подходит под тему "конверсии".
